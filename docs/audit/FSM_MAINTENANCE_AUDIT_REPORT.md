### دورة الحياة الكاملة:
```
pending → assigned → in_progress → on_hold → completed → posted_gl → closed
```
لكن `completed` ≠ `closed` (الإغلاق المحاسبي). يجب استدعاء `post-gl` منفصلاً بعد الإكمال — هذا فصل صحيح يسمح بمراجعة قبل الترحيل.

## 8. التوصيات

1. **ربط قطع الغيار بالمخزون (أولوية قصوى)**: إضافة `product_id` و `inventory_item_id` إلى `service_request_costs`. عند إضافة تكلفة نوعها `parts`، خصم الكمية من المخزون فوراً وإنشاء حركة مخزون (Stock Movement).

2. **بناء نظام الصيانة الوقائية**: إنشاء جدول `pm_plans` (خطة صيانة وقائية) مع: الأصل/المعدة، التكرار (شهري/ربع سنوي)، آخر تنفيذ، التالي. مجدول يومي يفحص الخطط ويُنشئ `service_request` تلقائياً عند استحقاق الموعد.

3. **إضافة SLA لأوامر الخدمة**: إضافة حقول `sla_response_hours`, `sla_resolution_hours`, `first_response_at`, `sla_status` إلى `service_requests`. مجدول يفحص الأوامر المتأخرة ويُرسل تنبيهات.

4. **بناء محرك فوترة الخدمات**: صفحة "إغلاق وفوترة" تحتسب تلقائياً: إجمالي قطع الغيار + (ساعات العمالة × سعر الساعة) + هامش ربح %. تعرض ملخصاً قبل الترحيل.

5. **تفعيل القفل المتفائل**: تنفيذ مقارنة `version` فعلياً في `update_service_request` لمنع تعارض التحديثات المتزامنة.

6. **إثراء نموذج الفني**: إضافة أدوار "فني ميداني"، مهارات، مناطق تغطية، وتقويم مواعيد.

7. **دمج أنظمة الصيانة الثلاثة**: حالياً يوجد 3 أنظمة صيانة منفصلة: `service_requests` (خدمة عملاء)، `asset_maintenance` (أصول ثابتة)، `maintenance_logs` (معدات تصنيع). توحيدها تحت مظلة واحدة.
