## السيناريو 1: محلي مرتبط بالسيرفر

### الفكرة الأساسية

```
┌─────────────────────────────────────────────────────────┐
│                                                           │
│  تطبيق ديسك توب (Electron/Native)                      │
│  ├─ واجهة بنفس تصميم 🌐 Web App                         │
│  ├─ SQLite محلي كـ "Cache" فقط                          │
│  └─ API Client → يتصل بـ Backend القديم                │
│                                                           │
│                    ↕ HTTP/REST API                       │
│                                                           │
│  ☁️ AMAN Backend (Python FastAPI)                       │
│  ├─ PostgreSQL (قاعدة البيانات الرئيسية)              │
│  ├─ Redis (جلسات + تخزين مؤقت)                         │
│  └─ Authentication (JWT Tokens)                          │
│                                                           │
└─────────────────────────────────────────────────────────┘
```

### المعمارية التفصيلية

#### المستوى 1: واجهة المستخدم (UI)
```
Desktop App (Built with Electron/Tauri/Qt)
├─ Widgets (نفس React components)
├─ Forms (تعديل البيانات محلياً)
├─ Reports (من SQLite المحلي)
├─ Dashboard (بيانات مزامنة)
└─ Status Indicator (⚪ Online / 🔴 Offline)
```

#### المستوى 2: طبقة التطبيق (Application Layer)
```
├─ Redux Store (حالة التطبيق)
│  ├─ Online Queue (عمليات معلقة للإرسال)
│  ├─ User Data (معلومات المستخدم)
│  └─ UI State (حالة الواجهة)
│
├─ Service Worker / Background Process
│  ├─ 🔄 Offline Detection
│  ├─ 🔁 Auto-Sync (كل 5 دقائق مثلاً)
│  ├─ ⚠️ Conflict Resolution
│  └─ 💾 Local Storage Manager
│
└─ IPC Bridge (الاتصال مع العملية الرئيسية)
```

#### المستوى 3: طبقة البيانات المحلية (Local Data)
```
SQLite Database (محلي)
├─ users (الموظفين النشطين)
├─ invoices (الفواتير المشتراة/المباعة)
├─ inventory (حركات المخزون)
├─ transactions (المعاملات المالية)
├─ documents (الملفات المرفقة)
├─ sync_queue (العمليات المعلقة)
├─ sync_log (تاريخ المزامنة)
└─ last_sync_timestamp (وقت آخر مزامنة)
```

#### المستوى 4: طبقة الشبكة (Network Layer)
```
HTTP Client
├─ Request Handler
│  ├─ ✅ Online: أرسل مباشرة
│  └─ ❌ Offline: أضف إلى Queue
│
├─ Response Handler
│  ├─ 200: تحديث SQLite
│  ├─ 401: Refresh Token
│  ├─ 4xx: عرض خطأ للمستخدم
│  └─ 5xx: أعد المحاولة بعد دقيقة
│
└─ Sync Engine
   ├─ Pull: تحميل آخر البيانات من السيرفر
   ├─ Push: إرسال التعديلات المحلية
   └─ Merge: دمج التغييرات (Last-Write-Wins)
```

### آلية العمل

#### 1️⃣ الحالة Online (متصل بالإنترنت)
```
المستخدم يكتب بيانات
        ↓
يضغط "حفظ"
        ↓
التطبيق يرسل مباشرة للـ Server
        ↓
Server يرد بـ 200 OK
        ↓
يُحدّث SQLite محليًا ✅
        ↓
ظهور رسالة "تم الحفظ"
```

#### 2️⃣ الحالة Offline (بدون إنترنت)
```
المستخدم يكتب بيانات
        ↓
يضغط "حفظ"
        ↓
التطبيق يتحقق: هل متصل؟ ❌
        ↓
يحفظ في SQLite محليًا فقط
        ↓
يضيف السجل في sync_queue
        ↓
ظهور رسالة "سيتم مزامنته عند الاتصال"
        ↓
[عند عودة الإنترنت]
        ↓
التطبيق يكتشف الاتصال تلقائياً
        ↓
يرسل جميع الأوامر من Queue بالترتيب
        ↓
يحدّث الحالة (✅ تم المزامنة)
```

### البيانات المخزنة محلياً

#### ماذا نخزن؟
- ✅ **جداول قراءة** (Ref Data): جدول الحسابات، المنتجات، الأطراف، الأقسام
- ✅ **كشوف البيع/الشراء**: الفواتير، الأوامر، الجلسات
- ✅ **عمليات محلية**: مسودات، عمليات قيد الإجراء
- ✅ **ملفات تعريف**: صور، مرفقات (مع حد أقصى 100 MB)

#### ماذا لا نخزن؟
- ❌ **بيانات سرية**: كلمات المرور (tokens فقط مؤقتة)
- ❌ **جميع البيانات**: لا نسخ كاملة من البيانات الضخمة
- ❌ **بيانات المستخدمين الآخرين**: فقط من لديهم صلاحيات

### حل التضاربات (Conflict Resolution)

#### السيناريو: تعديل نفس الفاتورة من جهازين

```
Laptop (محلي):                     Server:
-------------------                --------
الفاتورة #100                      الفاتورة #100
Amount: 1000                       Amount: 500 ← تم التحديث أثناء غياب الإنترنت
Offline Edit: +2000                Status: Approved
        ↓                                  ↓
عند العودة للإنترنت               
        ↓
يكتشف Conflict تلقائياً
        ↓
سياسة الحل: Last-Write-Wins
        ↓
الخادم: Amount = 500 (الأقدم)
المحلي: Amount = 1000 (الأحدث)
        ↓
الحل: تحديث المحلي بقيمة السيرفر
        ↓
إظهار تنبيه: "تم تحديث الفاتورة من السيرفر. تعديلاتك تم отменا"
```

## السيناريو 2: محلي غير مرتبط (Standalone)

### الفكرة الأساسية

```
┌──────────────────────────────────────┐
│                                        │
│  تطبيق ديسك توب (مستقل تماماً)      │
│  ├─ كود Backend مضمن (Python)      │
│  ├─ قاعدة بيانات محلية (SQLite)    │
│  ├─ بدون اتصال بالإنترنت           │
│  └─ نسخة مبسطة من الميزات          │
│                                        │
│     🔒 معزول تماماً من الإنترنت     │
│                                        │
└──────────────────────────────────────┘
```

### المعمارية التفصيلية

#### المستوى 1: واجهة المستخدم
```
Desktop App (Electron/Tauri/Qt Native)
├─ UI Forms (نفس التصميم)
├─ Reports (من SQLite المحلي فقط)
├─ Dashboard (لا يوجد بيانات خارجية)
└─ Menu Bar
   ├─ File (New, Open, Save, Export)
   ├─ Edit (Undo, Redo)
   ├─ Tools (Backup, Restore)
   └─ Help
```

#### المستوى 2: Backend مضمن (Embedded)
```
Built-in Python Server (SQLite + FastAPI محلي)
├─ HTTP Server يستمع على localhost:5000 فقط
├─ API Endpoints (نفس الـ REST API)
├─ Business Logic (حسابات، تقارير، معالجة)
└─ Authentication (محلي فقط)
```

#### المستوى 3: قاعدة البيانات
```
SQLite Database (ملف واحد)
├─ company (شركة واحدة من البداية)
├─ users (الموظفين)
├─ invoices (الفواتير)
├─ inventory (المخزون)
├─ accounts (الحسابات المالية)
├─ transactions (المعاملات)
└─ reports_cache (تخزين مؤقت للتقارير)
```

#### المستوى 4: نقل البيانات

```
Export/Import Options:
├─ 📊 CSV Export (للمراجعة في Excel)
├─ 📦 ZIP Backup (نسخة احتياطية كاملة)
├─ 📄 PDF Reports (طباعة الفواتير)
├─ 🔗 USB Sync (نقل بين أجهزة)
└─ ☁️ Cloud Upload (Optional - لنسخ احتياطي)
```

### آلية العمل

#### المسار البسيط

```
المستخدم يفتح التطبيق
        ↓
تحميل SQLite من ~/AMAN_STANDALONE/data.db
        ↓
بدء FastAPI Server على localhost:5000
        ↓
عرض الواجهة (تتصل بـ localhost)
        ↓
كل شيء يحدث محلياً 100%
        ↓
عند الإغلاق
        ↓
إيقاف السيرفر
        ↓
حفظ البيانات تلقائياً
```

### قيود Standalone

#### كل ما لا يمكن فعله:
```
❌ مزامنة مع أجهزة أخرى
❌ وصول متعدد المستخدمين من شبكة
❌ النسخ الاحتياطي المركزي
❌ التقارير المالية الحكومية (بدون توافق)
❌ تكامل مع بوابات الدفع الخارجية
❌ رسائل البريد الإلكتروني المتقدمة
```

#### ما يمكن فعله:
```
✅ إدارة منتجات محلية
✅ فواتير بيع/شراء محلية
✅ تقارير أساسية (حسابات، مخزون)
✅ حسابات مالية محلية
✅ نسخ احتياطية يدوية
✅ تصدير لـ Excel/PDF
✅ استخدام متعدد الحسابات (نفس الجهاز)
```

## الخلاصة

### المميزات الرئيسية

#### 🌐 مرتبط بالسيرفر:
```
✅ بيانات موحدة + مركزية
✅ متعدد المستخدمين + تعاون
✅ سجل عمليات شامل (Audit Trail)
✅ يعمل في الإنترنت البطيء (Sync تدريجي)
⚠️ يحتاج إنترنت دائماً (وإن كان بطيئاً)
```

#### 🖥️ Standalone:
```
✅ لا يحتاج إنترنت نهائياً
✅ سريع جداً على الجهاز
✅ لا يحتاج hosted server (توفير)
✅ كل شركة صغيرة تستطيع تشغيله
⚠️ بدون تعاون متعدد المستخدمين
⚠️ لا بيانات مركزية
```

### الخطوة التالية

```
يجب اختيار:

1️⃣ أي سيناريو نبدأ به؟
   → مرتبط = أولوية إذا الحجم كبير
   → Standalone = أولوية للـ affordability

2️⃣ أي framework للـ UI؟
   → Electron = أسهل (React موجود أصلاً)
   → Tauri = أخف وأسرع

3️⃣ هل نبني الاثنين معاً أم متسلسل؟
   → مشترك = 60% من الكود
   → منفصل = بدء الأول ثم الثاني
```

**انتظر تغذية راجعة لاختيار المسار الأول! 🚀**












لقد تم رفع البانات على السيرفر وايا الاكشنز يعمل بالشكل الصحيح ولكن لم يحدث السيرفر بالتحديثات الاخيرة 
لماذا هذا دائما يحصل
سأتحقق بسرعة من مسار النشر الفعلي في ci.yml وملفات docker-compose للتأكد من سبب تكرار المشكلة (غالبًا mismatch بين ما يُبنى وما يُخدّم أو كاش index.html).

سأفحص نقطتين أخيرتين سريعتين: هل يوجد Service Worker يثبت نسخة قديمة، وهل قد يكون عندك Nginx خارجي يقدّم مسار مختلف عن الحاوية. هذا يحدد سبب التكرار بدقة.

السبب يتكرر عندك بسبب مزيج 3 عوامل موجودة فعلًا في مشروعك:

مساران مختلفان لتقديم الـ Frontend
في النشر عندك يوجد مسار حاوية frontend (docker compose ... up -d --force-recreate frontend).
وفي نفس الوقت الـ workflow فيه مزامنة اختيارية إلى /var/www/aman/frontend إذا كان موجود.
هذا يعني أحيانًا التحديث يُبنى في الحاوية، لكن الطلبات الفعلية من الدومين قد تُخدَّم من مسار مختلف.
كاش index.html في طبقة Nginx الخارجية (أو البروكسي)
nginx.conf مضبوط جيدًا على no-store لـ index.html.
لكن production.conf (الذي يخدم من root /var/www/aman/frontend) لا يحتوي قواعد no-store لـ index.html و/.
النتيجة: المتصفح أو أي proxy يحتفظ بـ index.html قديم، فيستمر يطلب ملفات JS القديمة حتى بعد نجاح الـ deploy.
نجاح Actions لا يعني أن الدومين يقدّم نفس الـ SHA
