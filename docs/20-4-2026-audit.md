# 1. Critical Financial Risks

## FIN-C1 — Idempotency race window in `create_journal_entry`
**Location:** `backend/services/gl_service.py:52-61` and `:89-99`
**Description:** The idempotency and source-duplicate guards do `SELECT ... LIMIT 1`, then later `INSERT`. There is no unique DB constraint on `idempotency_key` or on `(source, source_id, entry_date)`. Two concurrent requests with the same idempotency key both pass the SELECT (empty) and both INSERT → duplicate JE, double posting to the GL, out-of-balance sub-ledger vs GL.
**Impact:** Double-booked revenue/expense, inflated AR/AP, trial-balance drift that is extremely hard to trace after the fact.
**Fix:**
```sql
ALTER TABLE journal_entries
  ADD CONSTRAINT uq_je_idempotency UNIQUE (idempotency_key);
CREATE UNIQUE INDEX uq_je_source ON journal_entries (source, source_id, entry_date)
  WHERE source <> 'Manual' AND source_id IS NOT NULL;
```
Then in `gl_service.py`, wrap the INSERT in a `try/except IntegrityError` that re-reads by key and returns the existing JE.

## FIN-C2 — Missing DB-level double-entry invariants
**Location:** `backend/models/domain_models/core_accounting.py`; Alembic migrations.
**Description:** The Python-level check in `gl_service.py:83` is the only guard. `journal_lines` has no `CHECK (debit >= 0 AND credit >= 0 AND NOT (debit > 0 AND credit > 0))`. No deferred constraint trigger on the header enforcing `Σ debit = Σ credit` per `journal_entry_id`. Any other path (manual SQL, migrations, imports, a future router) can persist unbalanced or negative postings.
**Impact:** Silent GL corruption. Trial balance won't tie.
**Fix:**
```sql
ALTER TABLE journal_lines
  ADD CONSTRAINT chk_jl_nonneg CHECK (debit >= 0 AND credit >= 0),
  ADD CONSTRAINT chk_jl_exclusive CHECK (NOT (debit > 0 AND credit > 0)),
  ADD CONSTRAINT chk_jl_nonzero  CHECK (debit + credit > 0);

CREATE CONSTRAINT TRIGGER trg_je_balanced
  AFTER INSERT OR UPDATE ON journal_lines
  DEFERRABLE INITIALLY DEFERRED
  FOR EACH ROW EXECUTE FUNCTION assert_je_balanced();
-- function raises if SUM(debit) != SUM(credit) for the JE (within 0.005)
```

## FIN-C3 — Closed-period check only in one code path
**Location:** `backend/services/gl_service.py:102-110` (good), but raw `INSERT INTO journal_entries` / `journal_lines` statements appear in:
- `backend/utils/treasury_gl.py`
- `backend/routers/finance/reconciliation.py` (reversal paths)
- `backend/routers/finance/checks.py`
- `backend/routers/finance/notes.py`
- `backend/routers/finance/intercompany.py` and `intercompany_v2.py`

**Description:** These bypass the centralized service and thus the fiscal-lock check. A posted JE can land in a closed fiscal period.
**Impact:** Violates period-closing controls; restated financials; audit failure (cannot demonstrate period integrity).
**Fix:** (a) route every posting through `gl_service.create_journal_entry`; (b) add a `BEFORE INSERT` trigger on `journal_entries` / `journal_lines` that rejects rows whose date falls in a closed `fiscal_periods` row when `status='posted'`.

## FIN-C4 — Posted JEs can be mutated
**Location:** no negative guard (trigger) on `journal_entries`/`journal_lines`; update statements in:
- `backend/routers/finance/accounting.py` (edit-draft path — accepts id without status re-check in one branch)
- migrations and fix scripts (`backend/fix_db.py`)

**Description:** Application layer hides the "edit" button for posted entries (`JournalEntryList.jsx:52`), but the API route is still registered and the backend does not re-verify `status='draft'` before `UPDATE` in all branches. There is no DB trigger to prevent `UPDATE journal_entries SET ... WHERE status='posted'`.
**Impact:** Fraud vector; audit-trail gap ("before" value lost if no prior audit log); restatement of posted financials without a counter-entry.
**Fix:**
```sql
CREATE FUNCTION block_posted_je_changes() RETURNS trigger AS $$
BEGIN
  IF OLD.status = 'posted' AND TG_OP IN ('UPDATE','DELETE') THEN
    IF NEW.status = 'void' AND OLD.status = 'posted' THEN RETURN NEW; END IF;
    RAISE EXCEPTION 'Posted journal entries are immutable';
  END IF;
  RETURN NEW;
END$$ LANGUAGE plpgsql;
CREATE TRIGGER trg_je_immutable BEFORE UPDATE OR DELETE ON journal_entries
  FOR EACH ROW EXECUTE FUNCTION block_posted_je_changes();
-- same pattern for journal_lines joined to posted headers
```

## FIN-C5 — `ON DELETE CASCADE` destroys audit history
**Location:** `backend/models/domain_models/core_accounting.py` — FK from `journal_lines.journal_entry_id` is `CASCADE`; FKs from `journal_lines.account_id` in several models lack `ON DELETE RESTRICT`.
**Description:** Deleting a JE silently removes its lines (no soft-delete, no tombstone). Deleting an account can orphan/destroy posted history.
**Impact:** Permanent audit-trail loss. Regulatory non-compliance.
**Fix:** Change to `ON DELETE RESTRICT`. Forbid hard-delete of `journal_entries`; implement void via reversal entry only. Protect `accounts` with `RESTRICT`.

## FIN-C6 — `float` in money arithmetic
**Location:**
- Frontend: pervasive `parseFloat(...).reduce((s,l)=>s+l, 0)` in `pages/Accounting/TrialBalance.jsx:92-96`, `IncomeStatement.jsx:83-84`, `BalanceSheet.jsx:74`, `Sales/InvoiceForm.jsx:252-277`, `ReceiptForm.jsx:220-272`, and ~40 more places.
- Backend: `backend/routers/reports.py` aggregates with Python `float` in some sums; `backend/utils/accounting.py:123` signature still accepts floats and converts, but callers pass float — the intermediate serialization round-trips lose precision (`float(debit_base)` in `gl_service.py:174`).
**Description:** `journal_lines` are inserted with `float(debit_base)` — Decimal→float cast loses precision for values > 15 significant digits (rare but possible) and for denormal sums (0.1 + 0.2). Frontend totals can drift from the server.
**Impact:** Reports display different totals than the GL. Line-sum audits fail (a 0.01 residue in a 10M-line company).
**Fix:** Bind Decimal (psycopg2 adapter) instead of `float(...)` in SQL params. On frontend use `decimal.js` or work in minor units (integers). Always show server-returned totals, never client-side recomputed totals, for authoritative reports.

## FIN-C7 — Multi-currency posting can double-convert
**Location:** `backend/services/gl_service.py:148-156`
**Description:** Each line's `line_rate` defaults to the header `exchange_rate`; `input_debit` and `input_credit` are already in line currency, multiplied by `line_rate` to get base. But `update_account_balance` passes both `debit_base`/`credit_base` **and** `debit_curr`/`credit_curr` — the foreign balance is updated only if `line_currency == account.currency`, which is correct. However:
- There is no validation that `header exchange_rate` and per-line `exchange_rate` are consistent (one JE may mix rates).
- When `line.amount_currency` is provided, it overrides the computed amount — but input `debit`/`credit` are still multiplied by rate. If caller passes both, results diverge silently.
- FX gain/loss is not auto-computed on settlement of foreign-currency AR/AP; must be handled in each module and isn't everywhere.

**Impact:** FX-revaluation errors; silent over/under-statement of foreign-denominated balances.
**Fix:** Reject JEs where header.exchange_rate differs from any line.exchange_rate for the same currency. Add a dedicated FX-gain/loss service invoked on every AR/AP settlement that crosses rate dates. Add periodic FX revaluation JEs.

## FIN-C8 — Multiple `is_base = TRUE` currencies possible
**Location:** `backend/models/domain_models/finance_currency_bank.py` — no partial unique index.
**Description:** Two rows can both be `is_base = TRUE`. `get_base_currency()` uses `LIMIT 1` — picks one arbitrarily. Rate math breaks for the other.
**Impact:** Accounts denominated in the "other" base currency post with wrong sign/amount.
**Fix:**
```sql
CREATE UNIQUE INDEX uq_currency_one_base ON currencies ((TRUE)) WHERE is_base = TRUE;
```

## FIN-C9 — Overlapping fiscal periods
**Location:** `backend/models/domain_models/shared_dashboard_fiscal_intercompany.py` (fiscal_periods).
**Description:** No exclusion constraint preventing two periods covering the same date. `is_closed` check on a transaction date returns the first match; if two periods overlap (one closed, one open), behavior depends on index order.
**Impact:** Can post into "open" period that overlaps a "closed" period → de facto reopened period.
**Fix:**
```sql
CREATE EXTENSION IF NOT EXISTS btree_gist;
ALTER TABLE fiscal_periods
  ADD CONSTRAINT excl_fiscal_no_overlap
  EXCLUDE USING gist (company_id WITH =, daterange(start_date,end_date,'[]') WITH &&);
```

# 3. Security Vulnerabilities

## SEC-C1 — Cross-tenant privilege escalation via `company_id` query param
**Location:** `backend/routers/roles.py:484, 555, 592, 627, 677, 740`; `backend/routers/audit.py:43, 192, 220`
**Description:** Endpoints accept optional `?company_id=` and fall back to the caller's tenant only when absent. RBAC check only verifies the caller has the permission in *their* tenant; never checks that `target_company_id == current_user.company_id`. Any admin of tenant A can manage/read tenant B.
**Attack:** `GET /api/audit/logs?company_id=<victim>` → victim audit log. `POST /api/roles/?company_id=<victim> {...}` → backdoor role in victim. `DELETE /api/roles/{id}?company_id=<victim>` → destroys victim RBAC.
**Fix:** Remove the parameter; derive tenant from JWT only. Gate any true cross-tenant need behind a distinct `superuser` permission validated against `system_companies` membership.

## SEC-C2 — Vertical escalation via HR update
**Location:** `backend/routers/hr/core.py:452-454`
**Description:** HR update-employee endpoint writes `company_users.role` directly from the payload with no separate `admin.roles` check, no rank check, no role-existence validation.
**Attack:** `PATCH /api/hr/employees/<self> { role: "admin" }` → tenant takeover on next token refresh.
**Fix:** Remove role-writing from HR endpoint. Move to `/api/roles` gated by `admin.roles`, validate role exists, enforce rank ≤ caller.

## SEC-C3 — Tenant DB role created with the tenant admin's application password
**Location:** `backend/database.py:164-167`
**Description:** On company registration `CREATE USER aman_<cid> WITH PASSWORD :admin_password` with `ALL PRIVILEGES` + `OWNER`. Attacker who registered knows the DB credential; any network path to PG bypasses app-level controls.
**Fix:** Generate a random 32-byte URL-safe password; store encrypted in `system_companies`. Or collapse to a single application PG user with schema-level isolation.

# 4. Data Integrity Issues

## DB-C1 — No unique constraint on `journal_entries.idempotency_key`
See FIN-C1. Critical.

## DB-C2 — No CHECK on `journal_lines` amounts
See FIN-C2. Critical.

## DB-C3 — `journal_entries.status` is free-form text
**Location:** `core_accounting.py`
**Description:** No CHECK or enum; any string accepted. Code-path asserts, DB does not.
**Fix:** `CHECK (status IN ('draft','posted','void','reversed'))` or PG enum type.

## DB-C4 — `ON DELETE CASCADE` propagates to posted financial history
**Location:** multiple FKs in `core_accounting.py`, `sales_customers_delivery.py`, `procurement_orders.py`
**Fix:** `ON DELETE RESTRICT` on every FK that points to a posted-history table. Use soft-delete or void flag instead.

## DB-C5 — FX rate history can be overwritten
**Location:** `finance_currency_bank.py` — `currencies.current_rate` updated in place; no `exchange_rate_history` table populated on update.
**Impact:** Historical JEs revalued inconsistently; can't reproduce past reports.
**Fix:** Immutable `exchange_rates(currency_id, effective_date, rate)` table with `UNIQUE (currency_id, effective_date)`.

## DB-C6 — Multiple `is_base = TRUE` possible
See FIN-C8.

## DB-C7 — Duplicate `account_code` per company possible
**Location:** `core_accounting.py` — `accounts` unique is global; should be scoped per `company_id` for the shared-schema model, or per-tenant DB (confirm the deployment model — code has both patterns).
**Fix:** `UNIQUE (company_id, account_code)` or unique per-tenant DB with a tenant-scoped migration harness.

## DB-C8 — Fiscal-period overlap possible
See FIN-C9.

## DB-C9 — `update_account_balance` fires for draft JEs in one code path
**Location:** `services/gl_service.py:182` guards `if status == "posted"`, but `utils/treasury_gl.py` and some direct inserts in `routers/finance/checks.py` call `update_account_balance` regardless of status.
**Fix:** Move balance-update into a trigger on `journal_lines` conditioned on the parent `status='posted'`, removing the application-level responsibility.

## DB-C10 — ORM `version` / optimistic-lock columns don't exist in DB
**Location:** `utils/optimistic_lock.py` references a `version` column on several tables that the Alembic baseline never creates.
**Impact:** Optimistic lock is a no-op silently; concurrent edits overwrite each other.
**Fix:** Alembic migration adding `version INTEGER NOT NULL DEFAULT 1` and wiring `__mapper_args__ = {"version_id_col": version}` on the models.

## DB-C11 — Alembic baseline is effectively empty; schema created by `Base.metadata.create_all()` at first boot
**Location:** `backend/alembic/versions/`
**Impact:** No migration discipline; schema drift per environment; can't audit historical schema changes.
**Fix:** Generate a consolidated baseline from the current metadata, then run every future change as a migration.

# 6. Architectural Improvements

## ARCH-C1 — Centralize all GL posting
**Problem:** Postings currently happen via `gl_service`, `utils/treasury_gl`, inline `INSERT` in finance routers. Fiscal lock, idempotency, audit, balance updates are each duplicated 3× and drift.
**Fix:** Make `gl_service.create_journal_entry` the **only** code path. Delete `treasury_gl` insert logic and have it build a lines list and call the service. Add a CI grep that fails the build on `INSERT INTO journal_entries` / `INSERT INTO journal_lines` outside `backend/services/gl_service.py`.

## ARCH-C2 — Enforce invariants at the DB, not in Python
**Problem:** Double-entry, period-lock, posted-immutability, balanced-JE are all application-level. The database is a passive store.
**Fix:** Triggers and constraints (FIN-C2, FIN-C4, DB-C3, DB-C10). Python is a thin layer over a self-consistent schema.

### ARCH-H
| ID | Issue | Fix |
|---|---|---|
| ARCH-H1 | Two intercompany modules (`intercompany.py`, `intercompany_v2.py`) coexist — divergent logic. | Delete v1; rename v2. |
| ARCH-H2 | Duplicate model trees (`models/core_accounting.py`, `models/domain_models/core_accounting.py`, `models/domains/finance.py`) — three sources of truth. | One canonical tree; remove the other two after migration. |
| ARCH-H3 | Background jobs in APScheduler in-process — unreliable. | Dedicated worker service. |
| ARCH-H4 | Reports computed from denormalized invoice/PO tables in some places, from GL in others. | Single rule: *financial* reports read GL, *operational* reports read transactional tables. Document. |
| ARCH-H5 | Mixed async/sync handlers — DB access still sync (`SQLAlchemy` core). | Pick one; if keeping sync DB, use `def` handlers only. |

### ARCH-M
- ARCH-M1: Per-tenant databases via PG role ownership — operational cost is high; consider schema-per-tenant.
- ARCH-M2: `database.py` uses raw text() everywhere; introduce repositories per aggregate to limit SQL sprawl.
- ARCH-M3: Pydantic v1 mixing with v2 patterns in `schemas/` — pick one.
- ARCH-M4: No event bus — modules call each other synchronously (e.g., invoice → GL → stock → notification). Introduce domain events.
- ARCH-M5: Tests don't assert GL balance after every business flow — add property-based tests (`hypothesis`) that random-walk flows and assert `Σ debit = Σ credit`.
- ARCH-M6: i18n `dir="rtl"` hardcoded in several components (`Sidebar.jsx:110`, `Topbar.jsx:133`, `CostingPolicy.jsx:156`) — breaks English.
- ARCH-M7: Frontend stores money as JS `number` — switch to `decimal.js` or minor-units integers.
