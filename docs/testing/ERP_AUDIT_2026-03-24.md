# تقرير تدقيق نظام ERP - 2026-03-24

## 1. نطاق التدقيق والمنهجية
يغطي هذا التدقيق سلوك الـ backend والتكاملات في نظام AMAN ERP عبر:
1. اختبارات API التي تم توليدها وتشغيلها بواسطة TestSprite.
2. تحقق تشغيلي مباشر على البيئة المحلية (`localhost:8000`).
3. مراجعة على مستوى الكود لسلامة المحاسبة، الأمان، منطق الأعمال، الاعتمادية، وجاهزية النشر.

سياق التشغيل:
1. الـ Backend يعمل وبحالة صحية.
2. الـ Frontend يعمل.
3. Redis غير متاح في التشغيل المحلي (وضع fallback مفعل).

## 2. الملخص التنفيذي
الحالة العامة: **غير جاهز للنشر قبل المعالجة**.

أبرز المعيقات:
1. وجود بيانات دخول إنتاجية hardcoded في سكربتات النشر.
2. وجود Tokens وAPI Keys حساسة في مخرجات الاختبار.
3. فشل حرج في دورة المبيعات `O2C` عند إنشاء الفاتورة بسبب عدم تطابق schema (`payment_method`).
4. خلل نزاهة مالي في دورة المشتريات `P2P` يسمح بأن يتجاوز `paid_amount` إجمالي الفاتورة.
5. عدم تطابق عقد المصادقة في الاختبارات (النظام يتوقع `form-data` بينما بعض الحالات استخدمت JSON).
6. وجود نقاط بحاجة تعزيز في التفويض (Authorization) وتحصين رفع الملفات.

الخطر: اختراق مباشر للبنية التحتية.

2. تسرب بيانات اعتماد ضمن مخرجات الاختبار.
السجل: `testsprite_tests/tmp/config.json` يحتوي bearer token وAPI key.
الخطر: إعادة استخدام/تسريب بيانات حساسة.

3. مفتاح سري بصيغة ضعيفة/افتراضية في بيئة التشغيل.
السجل: `backend/.env`.

4. فشل حرج في دورة `Order-to-Cash` عند إنشاء فاتورة المبيعات.

### 4.2 <span style="color:#b58900; font-weight:700;">🟡 مهم (Important)</span>
1. حاجة إلى تدقيق أعمق لنطاق التفويض على قراءات by-id لغير المستخدمين ذوي الصلاحية العليا.
السجل: بعض المسارات تحتاج إثباتا تشغيليا إضافيا على أدوار non-superuser.
الخطر: احتمال كشف بيانات عبر الفروع عند غياب row-level filters الصارمة.

2. تحصين رفع الملفات غير مكتمل.
السجل: يوجد تحقق امتداد/حجم، ولا يوجد تحقق MIME/signature.
الخطر: رفع ملفات متنكرة ضارة.

3. انتشار نمط SQL الديناميكي.
السجل: استخدامات متعددة لـ `text(f"...")` مع أجزاء ديناميكية.
الخطر: قابلية SQLi مستقبلية إذا فشل allowlist.

4. الاعتماديات غير مثبتة بإصدارات ثابتة في backend.
الخطر: عدم استقرار الإصدارات وsupply-chain drift.

5. مزيج هجرة قواعد البيانات (manual + Alembic).
الخطر: schema drift وترتيب هجرات غير حتمي.

### 4.3 <span style="color:#0a7f2e; font-weight:700;">🟢 منخفض (Minor)</span>

## 5. ما يعمل بشكل صحيح
1. مسارات hashing/verifying لكلمات المرور تعمل عبر bcrypt.
2. فحوصات القفل المحاسبي (Fiscal Lock) موجودة في مسارات رئيسية.
3. إطار تدقيق العمليات (Audit Logging) موجود ومتكامل.
4. أنماط rollback منتشرة في المسارات الحساسة للمعاملات.
5. سكربت النسخ الاحتياطي موجود ويدعم retention/rotation.

4. الرواتب، الإهلاك، التسوية البنكية.
الوضع: الوحدات موجودة والاختبارات موجودة بالمستودع.
الفجوة: دليل تنفيذ شامل لهذه الجلسة غير مكتمل بسبب مشكلة المصادقة.

5. معادلة الميزانية وقائمة الدخل.
الوضع: توجد اختبارات نزاهة في المستودع.

الوضع: مسار الدفع يعمل وظيفيا.
الفجوة: تم تأكيد حالة تجاوز `paid_amount` لقيمة الفاتورة، ما يستلزم قيدا محاسبيا مانعا على مستوى التطبيق/قاعدة البيانات.

## 7. حالة التدقيق الأمني
1. SQLi وXSS.
إيجابي: استخدام واسع للـ parameterized queries ووجود security middleware.
فجوة: SQL الديناميكي وفلاتر النص العامة لا تغني عن allowlists صارمة.

2. المصادقة والجلسات.
إيجابي: token blacklist وrate limiting موجودان.
فجوة: يجب توحيد عقد الاختبارات مع form-based login وإدارة دورة حياة token.

3. RBAC والتفويض.
إيجابي: استخدام permission decorators واسع.
فجوة: إضافة row-level filters صارمة على كل مسارات by-id مع اختبارات إلزامية لأدوار non-superuser.

4. تسرب البيانات الحساسة.
النتيجة: تم تأكيد تسربات حرجة ضمن سكربتات/Artifacts.

5. CSRF.
الوضع الحالي: النظام يعتمد bearer token وليس cookie-session افتراضياً.
توصية: عند اعتماد cookies مستقبلاً، تفعيل SameSite + CSRF tokens.

## 8. الأداء والاعتمادية
1. المعاملات وrollback موجودة في أجزاء كبيرة من النظام.
2. توجد safeguards للتزامن في مسارات معينة (`FOR UPDATE SKIP LOCKED`).
3. غير مكتمل في هذه الجلسة: قياسات حمل كمية (1000+ سجل، مستخدمون متزامنون، memory profiling).
